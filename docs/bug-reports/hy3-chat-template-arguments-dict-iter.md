# Hy3-preview chat template: `'str object' has no attribute 'items'` when re-rendering tool_calls

**Target:** inferencerlabs (HuggingFace community discussion / model repo)
Likely repo: `inferencerlabs/Hy3-preview-MLX-9bit` and the parent FP weights.

## Summary

The chat template Jinja iterates `tool_call.function.arguments.items()`
under the assumption that `arguments` is a dict. Per the OpenAI spec
(`/v1/chat/completions` request/response shape), `arguments` is a
**JSON-encoded string** so that it round-trips losslessly through the
wire format.

When a client re-sends the conversation history that includes a previous
`assistant` message with `tool_calls`, the Jinja render crashes:

```
jinja2.exceptions.UndefinedError: 'str object' has no attribute 'items'
  File "<template>", line 150, in top-level template code
```

This breaks any multi-turn agent loop with tools on Hy3-preview, e.g.
Notion MCP search → result → final summary.

## Reproduction

```python
from transformers import AutoTokenizer

tok = AutoTokenizer.from_pretrained("inferencerlabs/Hy3-preview-MLX-9bit")

messages = [
    {"role": "user", "content": "weather in Paris?"},
    {"role": "assistant", "content": "", "tool_calls": [{
        "id": "call_1", "type": "function",
        "function": {
            "name": "get_weather",
            "arguments": "{\"city\": \"Paris\"}"   # ← string per OpenAI spec
        }
    }]},
    {"role": "tool", "tool_call_id": "call_1",
     "content": "{\"temp\":18,\"sky\":\"cloudy\"}"},
]

tools = [{"type":"function","function":{
    "name":"get_weather","description":"Get weather",
    "parameters":{"type":"object","properties":{"city":{"type":"string"}}}
}}]

tok.apply_chat_template(messages, tools=tools, add_generation_prompt=True,
                        tokenize=False)
# → UndefinedError: 'str object' has no attribute 'items'
```

If you change `arguments` to a dict (`{"city": "Paris"}` instead of the
JSON string), the template renders fine. But that breaks the OpenAI spec.

## Reference: OpenAI spec

> `arguments` — **string**. The arguments to call the function with, as
> generated by the model in JSON format. Note that the model does not
> always generate valid JSON, and may hallucinate parameters not defined
> by your function schema. Validate the arguments in your code before
> calling your function.

https://platform.openai.com/docs/api-reference/chat/object#chat/object-choices

## Suggested fix in the chat template

Two options:

### Option A — accept both types

```jinja
{%- if tool_call.function.arguments is string %}
  {%- set args = tool_call.function.arguments | from_json %}
{%- else %}
  {%- set args = tool_call.function.arguments %}
{%- endif %}
{%- for k, v in args.items() %}
  ...
{%- endfor %}
```

### Option B — always normalize to dict at the top of the template

```jinja
{%- set _args = (tool_call.function.arguments | from_json) if (tool_call.function.arguments is string) else tool_call.function.arguments %}
```

## Workaround we deployed

In our distributed runner (`scripts/runner.py`), we pre-normalize
messages before `apply_chat_template`:

```python
def _normalize_messages_for_template(messages):
    for msg in messages:
        for call in (msg.get("tool_calls") or []):
            fn = call.get("function") or {}
            args = fn.get("arguments")
            if isinstance(args, str):
                try:
                    fn["arguments"] = json.loads(args) if args.strip() else {}
                except Exception:
                    fn["arguments"] = {}
    return messages
```

This unblocks Hy3-preview but pushes spec-violating data downstream, so
fixing the template upstream is preferable.
